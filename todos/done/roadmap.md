# v100: Research Roadmap & Strategic Vision
**Targeting the Frontier of Long-Horizon Agentic Systems**

## 1. Current State: The Observability Primitive
The v100 harness currently operates as a **High-Fidelity Epistemic Environment**. It treats the interaction between an LLM and a codebase not as a "chat," but as a series of **Observable State Transitions**.

### Core Technical Pillars:
*   **Trace-Driven Determinism:** By persisting every event (model call, tool I/O, compression) in a structured JSONL format, v100 enables **Offline Counterfactual Analysis**. We can "rewind" a failed run and test if a different model or a more precise tool result would have averted a "Tool Loop" or "Context Collapse."
*   **Hybrid Provider Economics:** v100 is uniquely positioned to bridge the gap between expensive API-driven research and high-volume experimentation via **Subscription-Backed Providers** (Codex/Gemini Pro). This allows for deep, 100+ step runs that would be cost-prohibitive on standard pay-per-token models.
*   **Primitive Multi-Agent Coordination:** The current `orchestrate` and `blackboard` implementations provide a foundation for **Stochastic Swarms**, where agents share a mutable, global state buffer (`blackboard.md`) to synchronize across asynchronous tasks.

## 2. Strategic Vision: Toward Cognitive Autonomy
The goal of v100 is to evolve from a passive observer into a **Meta-Cognitive Controller** that actively manages the agent's reasoning trajectory.

### A. From Trace to Policy (The Synthesis Loop)
The traces generated by v100 are not just logs; they are **Expert Trajectories**. By scoring these traces, we create a high-density dataset for **Direct Preference Optimization (DPO)**. v100 should eventually function as a "distillation furnace," where successful traces from SOTA models (e.g., GPT-5.3) are used to fine-tune local models (e.g., Qwen-3.5) specifically for the v100 toolset.

### B. Heterogeneous Cognitive Swarms
Real-world engineering requires a mix of "Fast" and "Slow" thinking. The vision is to enable **Tiered Orchestration**:
*   **Tier 1 (Local/Fast):** Ollama-based models handling routine "Read/List" operations.
*   **Tier 2 (Cloud/Deep):** Codex/Gemini-based models handling architectural decisions and complex refactors.
*   **Tier 3 (Meta-Agent):** A supervisor agent that monitors the `efficiency_score` and `latency_p95` in real-time, swapping models dynamically if "Tool Thrashing" is detected.

### C. Semantic Grounding
Agents currently view code as "text blocks." The strategic move is toward **Graph-Based Agency**, where tools like `sem_read` and `sem_impact` allow the agent to navigate a project's Abstract Syntax Tree (AST) rather than its file system. This reduces token noise and forces the agent to reason about **Logical Entities** (classes, functions, dependencies) rather than lines of text.

## 3. The Roadmap: A Multi-Phase Evolution

### Phase 1: Epistemic Foundation (COMPLETED)
*   **Vectorized Blackboard:** Replaced the flat `blackboard.md` with a local vector database. Implemented `blackboard_store` and `blackboard_search`.
*   **Uncertainty-Aware Tooling:** Implemented a `reflect` step for dangerous tools with confidence-based rejection thresholds.
*   **Semantic Navigation (Tree-Sitter):** Integrated Tree-Sitter with `fs_outline` tool for Go files.

### Phase 2: Meta-Cognitive Supervision (In Progress)
*   **Solver Abstractions (2a COMPLETED):** Refactored the core loop into pluggable `Solver` implementations (`react`, `plan_execute`). Added checkpointing and backtracking support.
*   **Automated Root Cause Analysis (ARCA):** When the `metrics` engine detects a `tool_loop` or `context_collapse`, it should autonomously launch a **Diagnostic Replay**.
*   **Adversarial Benchmarking:** A new `v100 stress` command that introduces artificial "hallucinations" (e.g., a tool returns a slightly wrong file path) to measure how effectively an agent can recover using its own internal verification logic.
*   **Human-in-the-Loop (HITL) Nudging:** Enhance the TUI to allow researchers to inject "Cognitive Anchors"—short, high-priority instructions that persist across compressions—to guide agents out of reasoning dead-ends.

### Phase 3: High-Fidelity Research Sandboxing (COMPLETED)
*   **Persistent Sandbox Sessions (3a COMPLETED):** Materialize an isolated per-run workspace and keep a runtime session alive across the entire run.
*   **Virtual Path Mapping (3b COMPLETED):** Present a stable `/workspace` namespace while mapping host and sandbox paths bidirectionally.
*   **Reliable Recovery (3c COMPLETED):** Tie workspace snapshots to explicit mutation metadata and restore loop+filesystem state together.
*   **Hardened Runtime Policy (3d COMPLETED):** Add Docker-backed subprocess execution with explicit seccomp, network isolation, backend-aware doctor checks, and apply-back/export support.

### Immediate Dogfood Priorities (Observed 2026-03-08)
*   **Metadata Visibility in Operator Surfaces:** `run.start` now carries `model_metadata` (context size, pricing, free/paid hints), but `meta.json`, `v100 stats`, `v100 compare`, and `v100 query` still hide it. Surface those fields where operators actually inspect runs, or the new provider-grounding work stays trapped in traces.
*   **Bench-First Demo Path:** `DF-08 Bench Smoke` is now the strongest live demo because it reliably exercises provider comparison, tagging, replay, and measurement with bounded prompts. Promote bench-driven usage in the product flow instead of treating it as a secondary feature.
*   **Gemini Tool-Response Accounting Hardening:** A simple two-file read run still triggered `gemini: HTTP 400` complaining about mismatched function call/response parts. Harden the provider adapter so multi-tool turns cannot desynchronize the provider request format.
*   **Prompt/Tool Surface Consistency:** The system prompt still tells models to verify with `sh`, but the live tool surface for default sandbox runs did not expose `sh` at all. Either enable `sh` consistently or stop promising it in the core agent instructions.
*   **Finalize Sandbox Review on User Exit:** A sandboxed edit run produced real workspace changes and snapshots, but no `sandbox_apply_back.json` artifact on `user_exit`. Manual review artifacts should be finalized whenever a run exits with pending sandbox changes, not only on the clean success path.
*   **Classifier Convergence:** Unify `v100 metrics` and `v100 analyze` so the same trace cannot be simultaneously labeled `tool_loop` and `normal`. The research loop breaks down if the scoring surfaces disagree on the same evidence.
*   **Patch Hygiene & Clean Sandbox Diffs:** Harden `patch_apply` so model-generated patch envelopes do not fail on first attempt, suppress `.orig` backup files, and filter incidental `.git/index` churn from `sandbox_apply_back.json` artifacts.
*   **Local Delegation Under Sandbox Policy:** Decouple `agent` / `dispatch` / `orchestrate` from blanket network gating so sandboxed runs at `network_tier = "off"` can still spawn bounded local sub-agents when no external network is required.
*   **Synthesis Watchdogs:** Detect long inspection loops that burn tens of thousands of tokens without producing a final answer, then force a synthesis turn, tool-family cooldown, or early-stop classification before the run silently degrades.
*   **Large-Output Guardrails:** Cap, summarize, or reject oversized tool outputs before they are fed back into the provider context. A single binary read should not be able to trigger provider-side 400s because the transcript now contains a 20MB+ payload.
*   **Derived Trace Views & Failure Digests:** Give agents first-class access to compact per-run summaries (`stats`, `metrics`, `analyze`, last errors, retry clusters, artifact diffs) so recursive self-critique never depends on reading raw `trace.jsonl` blobs.
*   **Provider-Aware Evidence Budgeting:** Shape trace slices and tool outputs to each provider’s real payload limits. If Codex cannot safely ingest a raw trace or even a verbose summary, the harness should automatically downgrade to smaller evidence packets instead of discovering the limit via 400/413 failures.
*   **Workspace Salience Filters:** Teach the harness to down-rank compiled binaries and other generated artifacts during repo exploration so agents stop reading irrelevant files like the built `v100` executable when source files are the real target.
*   **Nightly Dogfood Pack:** Run the `dogfood/` tasks across Codex and Gemini on every meaningful harness change, and track success rate, token cost, retry rate, and sandbox artifact quality as release-blocking signals.

### Phase 4: Collective Intelligence & A2A Protocols
*   **Agent-to-Agent Mesh:** Implement standardized A2A protocols for cross-provider negotiation and shared memory handoffs.
*   **Consensus Solvers:** Implement multi-agent voting and judging to mitigate model-specific hallucinations.

### Phase 5: Autonomous Research & Goal Invention
*   **Active Goal Discovery:** The system scans the environment and autonomously proposes new tasks and research objectives.
*   **Autonomous Benchmarking:** v100 identifies failure clusters and launches its own ablation experiments to test strategy mutations.

### Phase 6: Neural Forging & Self-Distillation
*   **Self-Tuning Loop:** The agent uses successful trajectories to autonomously generate synthetic datasets for local fine-tuning.
*   **Model Specialization:** Agents benchmark 'Old Self' vs 'New Self' and autonomously update the model weights in their default Policy.

### Phase 100: Recursive Self-Evolution (The "Singularity")
*   **Autonomous Tool Synthesis:** A `tool_builder` role that can write and register new Go-based tool plugins in real-time.
*   **Self-Prompt Optimization:** Meta-optimizers that refine `system_prompts` based on empirical `v100 score` data.
*   **Substrate Migration (X100):** Agents that monitor their own latency and autonomously migrate their sandbox and trace to higher-compute hardware mid-trajectory.

## 4. Key Performance Indicators (Research Metrics)
To measure success, the system will track:
1.  **Tokens per Unit of Progress (TPUP):** How many tokens are consumed to achieve a "Pass" score.
2.  **Recovery Rate:** The percentage of "Normal" classifications after an initial "Tool Thrashing" detection.
3.  **Orchestration Depth:** The maximum number of successful sub-agent delegations without losing the original objective's "Strategic Intent."
