# v100 Phase 100 Specification: Recursive Self-Evolution

Phase 100 represents the "singularity" point for the v100 harness. It moves beyond observing agent behavior to enabling **Autonomous Capability Expansion**. In this phase, the agent is no longer just a user of the harness; it becomes the **Architect** of the harness.

---

## 1. Autonomous Tool Smithing (`tool_smith`)
**Objective:** Enable agents to identify gaps in their current toolkit and autonomously write, compile, and register new Go-based tools to bridge those gaps.

### Implementation Details:
*   **The `tool_smith` Specialist:** A dedicated agent role with a system prompt focused on Go software engineering and the v100 `Tool` interface.
*   **The Synthesis Workflow:**
    1.  **Gap Identification:** If an agent fails a task with a `no_progress` or `tool_error_cascade` classification, it dispatches to the `tool_smith`.
    2.  **Implementation:** The `tool_smith` writes a new `.go` file in `internal/tools/dynamic/`.
    3.  **Hot-Reloading:** v100 implements a dynamic plugin loader (using Go's `plugin` package or, more robustly, a `go build` + secondary process execution) that re-registers the tool registry without stopping the run.
*   **Safety Rails:** Dynamic tools are *always* marked as `Dangerous` and require manual `ConfirmFn` approval before their *first* execution in a run.

---

## 2. Meta-Cognitive Feedback Loops (Metrics-as-Input)
**Objective:** Allow the agent to "see" its own performance metrics and diagnostic classifications in real-time to trigger self-correction.

### Implementation Details:
*   **Diagnostic Injection:** After every 5 steps, the `internal/core/loop.go` runs a background `ComputeMetrics` and `ClassifyRun`.
*   **Self-Correction Prompt:** If the classification is anything other than `normal` (e.g., `tool_loop` or `context_collapse`), the harness injects a high-priority system message:
    *   *"SYSTEM ALERT: Trace analysis detects a 'tool_loop' pattern. You have called [TOOL_NAME] 3 times with similar arguments. Please pivot your strategy or use the 'reflect' tool to analyze the failure."*
*   **The `reflect` Tool:** A new core tool that forces the agent to output a "Strategy Revision" block, which is then pinned to the top of the conversation context, superseding previous plans.

---

## 3. Automated Distillation Pipeline (Golden Trace Generation)
**Objective:** Turn successful, high-horizon runs into structured training data for smaller, local models.

### Implementation Details:
*   **Trace-to-DPO Converter:** A new command `v100 distill <run_id>` that takes a "Pass" scored run and generates a pair of trajectories:
    *   **Chosen:** The successful sequence of tool calls and reasoning.
    *   **Rejected:** Synthetic "negative" examples generated by taking "Fail" scored runs with similar prompts.
*   **Fine-Tuning Integration:** Export format compatible with Unsloth or Axolotl for immediate fine-tuning of Llama-3 or Qwen-2.5 models.
*   **Goal:** Create a "v100-Small" model (7B-14B) that outperforms GPT-4o specifically within the v100 harness by internalizing the project's unique tool schemas and memory patterns.

## 4. Execution Plan (Phase 100)

1.  **Step 1: Dynamic Registry.**
    *   Refactor `internal/tools/registry.go` to support runtime registration of new tool instances.
    *   Create `internal/tools/dynamic/` directory for agent-generated code.
2.  **Step 2: Real-time Diagnostics.**
    *   Update `Loop.Step` to trigger `ClassifyRun` periodically.
    *   Implement the "System Alert" injection logic.
3.  **Step 3: The `distill` Command.**
    *   Implement `internal/eval/distill.go` to transform JSONL traces into ShareGPT or Alpaca formats.
4.  **Step 4: Self-Optimization Test.**
    *   Launch a run where the agent is explicitly tasked with "Improving its own ability to search SQL databases" and verify it can successfully create and use a new SQL-specific tool.
