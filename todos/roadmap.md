# v100: Research Roadmap & Strategic Vision
**Targeting the Frontier of Long-Horizon Agentic Systems**

## 1. Current State: The Observability Primitive
The v100 harness currently operates as a **High-Fidelity Epistemic Environment**. It treats the interaction between an LLM and a codebase not as a "chat," but as a series of **Observable State Transitions**.

### Core Technical Pillars:
*   **Trace-Driven Determinism:** By persisting every event (model call, tool I/O, compression) in a structured JSONL format, v100 enables **Offline Counterfactual Analysis**. We can "rewind" a failed run and test if a different model or a more precise tool result would have averted a "Tool Loop" or "Context Collapse."
*   **Hybrid Provider Economics:** v100 is uniquely positioned to bridge the gap between expensive API-driven research and high-volume experimentation via **Subscription-Backed Providers** (Codex/Gemini Pro). This allows for deep, 100+ step runs that would be cost-prohibitive on standard pay-per-token models.
*   **Primitive Multi-Agent Coordination:** The current `orchestrate` and `blackboard` implementations provide a foundation for **Stochastic Swarms**, where agents share a mutable, global state buffer (`blackboard.md`) to synchronize across asynchronous tasks.

## 2. Strategic Vision: Toward Cognitive Autonomy
The goal of v100 is to evolve from a passive observer into a **Meta-Cognitive Controller** that actively manages the agent's reasoning trajectory.

### A. From Trace to Policy (The Synthesis Loop)
The traces generated by v100 are not just logs; they are **Expert Trajectories**. By scoring these traces, we create a high-density dataset for **Direct Preference Optimization (DPO)**. v100 should eventually function as a "distillation furnace," where successful traces from SOTA models (e.g., GPT-5.3) are used to fine-tune local models (e.g., Qwen-3.5) specifically for the v100 toolset.

### B. Heterogeneous Cognitive Swarms
Real-world engineering requires a mix of "Fast" and "Slow" thinking. The vision is to enable **Tiered Orchestration**:
*   **Tier 1 (Local/Fast):** Ollama-based models handling routine "Read/List" operations.
*   **Tier 2 (Cloud/Deep):** Codex/Gemini-based models handling architectural decisions and complex refactors.
*   **Tier 3 (Meta-Agent):** A supervisor agent that monitors the `efficiency_score` and `latency_p95` in real-time, swapping models dynamically if "Tool Thrashing" is detected.

### C. Semantic Grounding
Agents currently view code as "text blocks." The strategic move is toward **Graph-Based Agency**, where tools like `sem_read` and `sem_impact` allow the agent to navigate a project's Abstract Syntax Tree (AST) rather than its file system. This reduces token noise and forces the agent to reason about **Logical Entities** (classes, functions, dependencies) rather than lines of text.

## 3. The Roadmap: A Multi-Phase Evolution

### Phase 1: Epistemic Foundation (COMPLETED)
*   **Vectorized Blackboard:** Replaced the flat `blackboard.md` with a local vector database. Implemented `blackboard_store` and `blackboard_search`.
*   **Uncertainty-Aware Tooling:** Implemented a `reflect` step for dangerous tools with confidence-based rejection thresholds.
*   **Semantic Navigation (Tree-Sitter):** Integrated Tree-Sitter with `fs_outline` tool for Go files.

### Phase 2: Meta-Cognitive Supervision (In Progress)
*   **Solver Abstractions (2a COMPLETED):** Refactored the core loop into pluggable `Solver` implementations (`react`, `plan_execute`). Added checkpointing and backtracking support.
*   **Automated Root Cause Analysis (ARCA):** When the `metrics` engine detects a `tool_loop` or `context_collapse`, it should autonomously launch a **Diagnostic Replay**.
*   **Adversarial Benchmarking:** A new `v100 stress` command that introduces artificial "hallucinations" (e.g., a tool returns a slightly wrong file path) to measure how effectively an agent can recover using its own internal verification logic.
*   **Human-in-the-Loop (HITL) Nudging:** Enhance the TUI to allow researchers to inject "Cognitive Anchors"—short, high-priority instructions that persist across compressions—to guide agents out of reasoning dead-ends.

### Phase 3: Recursive Evolution (The "Self-Improving" System)
*   **Autonomous Tool Synthesis:** A `tool_builder` role that can write new Go-based tool plugins for the v100 registry. If an agent repeatedly fails a task because it lacks a specific capability (e.g., "I need a tool to diff SQL schemas"), it dispatches a request to the `tool_builder` to expand the harness's own capability.
*   **Self-Prompt Optimization (DSPy Integration):** Use the `v100 score` feedback to automatically rewrite `system_prompts`. If agents frequently "thrash" on a certain task, the harness uses a "Meta-Optimizer" to refine the role definition until the `efficiency_score` improves.
*   **Cross-Provider Distillation:** A "Golden Trace" pipeline that automatically exports successful multi-turn trajectories into a format suitable for fine-tuning local models (SFT/DPO), creating a specialized "v100-LLM" that is natively optimized for the harness's specific tool/policy/memory dialect.

## 4. Key Performance Indicators (Research Metrics)
To measure success, the system will track:
1.  **Tokens per Unit of Progress (TPUP):** How many tokens are consumed to achieve a "Pass" score.
2.  **Recovery Rate:** The percentage of "Normal" classifications after an initial "Tool Thrashing" detection.
3.  **Orchestration Depth:** The maximum number of successful sub-agent delegations without losing the original objective's "Strategic Intent."
